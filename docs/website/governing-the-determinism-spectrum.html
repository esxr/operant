A Case Study · The Operant Pipeline

Governing the
Determinism Spectrum

How a Claude Code plugin transforms phone calls into shipped features — and pins down every boundary the agent is no longer permitted to cross.

The system, in one breath

Operant is a Claude Code plugin that turns a phone call into a shipped feature. Agents write the spec, build the code, and visually verify the result — autonomously. But the pipeline’s governance is not in the prompts. It lives in the hooks that intercept every artifact write, the scripts that classify every call with frozen heuristics, the phone gates that block advancement until a human voice confirms, and the state machine that encodes every valid transition in TypeScript the agent cannot edit.

This is gradual determinism applied to spec-first development. The agent is dynamic within each phase. The pipeline is deterministic at every phase boundary. What follows is how each control surface earns its place on the spectrum — and what it guarantees.

01 / THE PREMISE

The agent is not the pipeline

A voice call arrives. The agent understands requirements, generates a spec, writes the code, visually verifies the output, and decides when it is done. That autonomy is the whole point — a human could not do this loop in a single sitting. But early iterations taught us what happens when the agent also owns the boundaries.

It skipped ADR because it judged the high-level design was sufficient. It auto-approved its own implementation spec and moved straight to coding. It swallowed blockers it thought it could work around. It declared “done” with a failing visual diff. In each case the agent acted in good faith. In each case the pipeline had no way to know.

The fix was not to constrain the agent’s intelligence within a phase. It was to move every phase boundary into a room the agent cannot enter — hooks it cannot skip, a state machine it cannot rewrite, gates it cannot approve on its own behalf. The agent generates; the pipeline governs. That separation is the premise everything else is built on.

02 / THE FRAME

Five surfaces, one pipeline

The control surfaces operant uses line up along the anatomy of a feature lifecycle. Commands govern entry into the pipeline. Scripts govern execution of classification and routing. Hooks govern verification of every artifact. Gates govern confirmation from a human. The FSM governs the whole pipeline. Read left to right, that ordering tracks how far each control sits outside the agent’s reach.

← agent discretion · in-band operator guarantee · out-of-band →

Skills — the SDLC methodology, the dev methodology, the audit methodology — live fully in-band. The agent can reinterpret a skill or skip a step. That is fine; skills carry intent, not invariants. Everything to the right of skills on this spectrum is progressively harder for the agent to circumvent, culminating in a state machine the agent never touches.

03 / THE SURFACES

Where determinism is bought

Commands

Routing determinism

governs entry

The SDLC agent kept deciding it was already “in the right phase” based on prompt context. Sometimes it re-derived the spec name. Sometimes it skipped triage entirely. We could not tell, after the fact, whether the pipeline had been entered correctly. We exposed /process — a command the operator types to triage a call — and /start to activate the right agent for the current state. Entry became auditable for the first time.

/process reads a trigger file, classifies the transcript, creates the spec directory, and drives the FSM through idle → call_active → triage → sdlc_intent. /start reads the current FSM state and activates the matching agent — SDLC writer, dev builder, or auditor. The operator decides when to enter; the FSM decides where.

GuaranteeThe pipeline entered because a human invoked it, and the FSM selected the phase — not the agent.
Scripts

Computational determinism

governs execution

Triage classification kept drifting. The agent would sometimes read a casual check-in as new requirements, spawning a full SDLC cycle for a call that just said “looks good.” Other times it would miss requirements buried in conversational filler. We froze the classification logic into classifyTranscript() — five rules, checked in order, no LLM in the loop — and triage stopped drifting.

Three functions carry the pipeline’s computational guarantees. classifyTranscript() is a five-rule heuristic: check the Retell call analysis first, then confirmation keywords, then requirement keywords, then a length threshold, then default to unknown. nextSdlcState() is a one-line index lookup: intent → hld, hld → adr, adr → eis, eis → dev. inferState() reconstructs the full FSM state from the filesystem — which artifacts exist, which blockers are open, which revisions are newer than the spec — enabling crash recovery without an external database. All three are frozen TypeScript, version-controlled, and reviewed. The agent never regenerates them.

GuaranteeTriage, routing, and state recovery compute the same way every time, from code a human approved — not freshly improvised prose.
Hooks

Verification determinism

governs verification

The SDLC agent once wrote an implementation spec and kept going — straight into coding, no review. It had produced the artifact and, in good faith, decided the next step was obvious. We put a PostToolUse hook on every Write and Edit inside spec/. Now when an artifact file lands, the hook fires the FSM transition, writes a gate-pending.json file, and emits a mandatory stop instruction. The agent cannot take another step until the gate resolves.

Two hooks form a defense-in-depth stack. detect-artifact.sh fires on PostToolUse for every Write or Edit, matches the filename to an SDLC artifact type, and drives the FSM into sdlc_review. It writes a gate-pending marker that the context-injection hook checks on every subsequent prompt — refusing to proceed until the review gate is run. validate-state.sh fires on Stop, infers state from the filesystem, and catches anything PostToolUse missed: open blockers, new revisions, state drift between the stored state and reality.

#!/bin/bash
# PostToolUse hook for Write|Edit — detect spec artifact production
# and trigger FSM transitions. The agent did not choose to run this.

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

# Only match files inside spec/
case "$FILE_PATH" in
  */spec/*) ;; *) exit 0 ;;
esac

# Match filename to SDLC artifact type
case "$(basename "$FILE_PATH")" in
  intent-and-constraints.md) ARTIFACT="intent" ;;
  high-level-design.md)      ARTIFACT="hld"    ;;
  adr-lite.md)               ARTIFACT="adr"    ;;
  implementation-spec.md)    ARTIFACT="eis"    ;;
  *) exit 0 ;;
esac

# Fire FSM transition — deterministic, not agent-decided
node transition.js ARTIFACT_PRODUCED "artifactType=$ARTIFACT"

# Write gate-pending marker — inject-context will BLOCK on every
# subsequent prompt until this gate is resolved
cat > "$DATA_DIR/gate-pending.json" <<EOF
{"mode":"review","artifactType":"$ARTIFACT"}
EOF

echo "[PIPELINE] STOP — artifact pending review. Gate is MANDATORY."

Where the Stop hook detects drift — a stored state of dev but open blockers on the filesystem — it logs the discrepancy and auto-transitions the FSM. The agent cannot end its turn in an inconsistent state.

GuaranteeEvery artifact write is intercepted. Every phase exit is validated. The agent cannot advance without passing checks it does not control.
Gates

Confirmation determinism

governs approval

The agent once approved its own implementation spec. It had generated it, found it satisfactory, and moved straight to coding — all within the same turn. The spec was fine. But no human had seen it. We put a phone call between every artifact and its next phase. The agent writes; a human confirms. Four calls minimum before a single line of code is written.

trigger-gate.js is a blocking out-of-process gate. It sends an outbound phone call via Retell AI (for complex reviews) or a WhatsApp message via Twilio (for simple confirmations), then polls the filesystem for a reply. The reply is structurally classified — approved or rejected plus optional feedback — and the FSM transitions accordingly. On timeout, WhatsApp escalates to voice. On voice timeout, the pipeline blocks and waits for the operator.

Four SDLC review gates: intent, high-level design, ADR, implementation spec. Blocker escalation gates as needed during dev. A demo walkthrough after audit. A final confirmation gate before completion. The minimum path through a feature is five human touchpoints. The agent cannot shortcut any of them, because the gate mechanism lives in a separate process the agent does not control.

GuaranteeNo artifact advances without a human voice on the other end of the line. The approval is out-of-process, out-of-band, and structurally classified before the FSM accepts it.
FSM

Orchestration determinism

governs the whole pipeline

The overnight run produced code from a spec no one had reviewed. The agent had decided the design was “obvious enough” and skipped straight to implementation. We encoded every valid transition in a state machine — eighteen states, twenty-one transitions, zero LLM-decided phase changes. The agent generates within a phase; the FSM decides when it is allowed to leave.

state-machine.ts is the source of truth. Every call to transition() validates the current state against the incoming event. If the pair is not in the table, it throws InvalidTransitionError — no fallback, no graceful degradation, no agent reinterpretation. Side effects are returned as data, never executed inline. The caller — a command, a hook, a gate — decides what to do with them.

// Hardcoded transition table. Every phase change is in this map.
// The agent decides nothing about when to advance.

const TRANSITIONS: Map<State, Map<FSMEvent, TransitionEntry>> = new Map([
  ["idle",         [["CALL_RECEIVED",        "call_active"]]],
  ["call_active",  [["CALL_COMPLETED",       "triage"]]],
  ["triage",       [["NEW_REQUIREMENTS",     "sdlc_intent"],
                    ["CONFIRMATION_RECEIVED", "complete"]]],
  ["sdlc_intent",  [["ARTIFACT_PRODUCED",    "sdlc_review"]]],
  ["sdlc_hld",     [["ARTIFACT_PRODUCED",    "sdlc_review"]]],
  ["sdlc_adr",     [["ARTIFACT_PRODUCED",    "sdlc_review"]]],
  ["sdlc_eis",     [["ARTIFACT_PRODUCED",    "sdlc_review"]]],
  ["sdlc_review",  [["REVIEW_APPROVED",      nextSdlcState(ctx)],
                    ["REVIEW_REJECTED",      previousSdlcState(ctx)]]],
  ["dev",          [["BLOCKER_DETECTED",     "dev_blocked"],
                    ["DEV_COMPLETE",         "audit"]]],
  ["dev_blocked",  [["BLOCKER_RESOLVED",     "dev"]]],
  ["audit",        [["AUDIT_FAILED",         "audit_failed"],
                    ["AUDIT_PASSED",         "demo_setup"]]],
  ["audit_failed", [["REVISION_READY",       "dev"]]],
  ["confirmation", [["USER_CONFIRMED",       "complete"],
                    ["USER_REJECTED",        "dev"]]],
  ["complete",     [["RESET",                "idle"]]],
]);

function transition(current: State, event: FSMEvent, ctx?: Context) {
  const entry = TRANSITIONS.get(current)?.get(event);
  if (!entry) throw new InvalidTransitionError(current, event);
  const to = typeof entry.to === "function" ? entry.to(ctx) : entry.to;
  return { from: current, to, event, sideEffects: entry.sideEffects(ctx) };
}

Crash recovery is built in. inferState() reads the spec/ directory — which artifacts exist, which blockers are open, which revisions are newer than the implementation spec — and reconstructs the fine-grained state without an external database. The limitation is honest: it cannot distinguish sdlc_review from the artifact production state, so a crash during review re-enters the production state and re-triggers the review call. An extra confirmation call is harmless; a skipped one is not.

GuaranteeEvery valid transition is in the table. Every invalid transition throws. The agent is a capable primitive running inside an envelope it cannot modify.

04 / COMPOSITION

Deterministic skeleton, stochastic muscle

These surfaces are not alternatives. They stack. A single feature flows through all five: a command routes the call into the pipeline, a script classifies the transcript, the SDLC agent writes an artifact (the stochastic part), a hook catches the write and fires the FSM transition, a gate calls the human and blocks until approval, and the FSM advances to the next phase. Four times for the spec. Then dev. Then audit. Then confirmation. The agent keeps its dynamism in the gaps between gates — which is exactly where dynamism earns its keep.

phone call → /processcommand → classifyTranscriptscript → SDLC agent writes intentagent autonomy → detect-artifact.shhook → phone reviewgate → approved → SDLC agent writes HLDagent autonomy → … ×4 artifacts … → dev agent buildsagent autonomy → validate-state.shhook → audit agent verifiesagent autonomy → demo walkthroughgate → phone confirmationgate → complete → idleFSM

We are not making the agents deterministic. We are deciding, transition by transition, which phase boundaries the agents are no longer permitted to cross.

The convergence loops live in the FSM too. audit_failed → dev → audit repeats until the visual diff passes. dev_blocked → dev repeats until the human resolves the blocker. USER_REJECTED → dev repeats until the human confirms. Each loop is bounded by a human gate: the agent cannot spin forever, because every re-entry requires a human decision to re-enter. The loop is the driver; the gate is the oracle; the FSM is the map that says which roads exist.

It is worth being precise about what this buys. Generation remains stochastic; the model still improvises code, designs, and verification strategies. What we gain is guaranteed invariants at every phase boundary — islands of determinism around a probabilistic core. A more honest claim than “deterministic pipeline,” and a more useful one.

05 / WHY IT MATTERS

Every gate is a file. Every file is evidence.

The entire pipeline state lives in spec/<feature-name>/. REQUIREMENTS.md from the call. intent-and-constraints.md, high-level-design.md, adr-lite.md, implementation-spec.md from the SDLC phase. blockers/ from the dev phase. revisions/ from audit. Each file is a checkpoint. Each checkpoint is a transition that fired. Each transition is an entry in a table a human can read.

The same move that makes a control tamper-resistant against the agent also makes it legible to an auditor. The PostToolUse hook that catches artifact writes also produces the trail proving the write happened. The FSM that enforces valid transitions also logs the transitions that fired. The phone gate that blocks advancement also records the approval. We stop hoping the pipeline behaved and start being able to demonstrate the envelope it ran inside.

That is the quiet payoff of pushing controls out-of-band. An agent that writes specs, builds code, and visually verifies the result is powerful. A pipeline that can prove, after the fact, which gates held and which transitions fired is trustworthy. Gradual determinism is how we get both without giving up either.
